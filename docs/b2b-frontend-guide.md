# B2Bフロントエンド設計・実装ガイド

更新日: 2026-07-19

## 1. 適用範囲

本ガイドは、管理画面、業務システム、SaaS、社内ツールなど、法人・組織向けWebアプリケーションの設計と実装を対象とする。

B2Bフロントエンドでは、視覚的な新規性より次の品質を優先する。

- **業務の正確性:** 業務ルール、状態遷移、権限を正しく反映する。
- **操作効率:** 反復作業、検索、比較、一括操作、キーボード操作を効率化する。
- **予測可能性:** 操作結果、対象範囲、保存状態を明示する。
- **回復可能性:** 入力エラー、通信障害、競合、部分失敗から復旧できる。
- **監査可能性:** 誰が、いつ、何を変更したかを追跡できる。
- **アクセシビリティ:** 利用環境や身体特性にかかわらず主要業務を完了できる。
- **保守性:** 共通コンポーネント、型、テスト、設計ルールを再利用する。

## 2. B2Bプロダクトで考慮する条件

| 条件 | 設計上の要件 | 避ける実装 |
| --- | --- | --- |
| 同じ業務を繰り返す | キーボード操作、保存済みビュー、一括操作、適切な情報密度 | 見た目を優先して比較・操作効率を落とす |
| データ量が多い | 検索、絞り込み、並び替え、ページング、列設定 | 意味が曖昧な全選択、無目的な無限スクロール |
| 複数の役割がある | ロール・属性・対象データに基づく認可 | ボタンの非表示だけで認可を実現する |
| データが長期間保持される | 履歴、更新者、競合検知、再開地点 | 作成直後の正常系だけを設計する |
| 承認や例外処理がある | 状態遷移、理由、差し戻し、部分成功 | 正常系だけの直線的なフロー |
| 誤操作の影響が大きい | 対象範囲、影響、確認、取消・訂正、監査記録 | 対象や結果が不明な確認ダイアログ |
| 顧客ごとに設定が異なる | 契約プラン、機能設定、組織・テナント境界 | 条件分岐を画面内へ無秩序に散在させる |
| 国・地域をまたぐ | ロケール、タイムゾーン、通貨、照合順 | 日付や金額を表示用文字列として保存する |

## 4. 業務モデルと状態遷移

画面を設計する前に、業務対象と操作可能条件を定義する。

| 項目 | 確認内容 |
| --- | --- |
| エンティティ | 何を管理するか |
| 識別子 | 人とシステムが対象をどう特定するか |
| 所有者 | どの組織・テナント・利用者に属するか |
| 状態 | 下書き、申請済み、承認済み、無効など |
| 状態遷移 | 誰が、どの条件で、どの状態へ変更できるか |
| 不変条件 | 常に成立しなければならない業務ルール |
| 副作用 | 通知、請求、外部連携、監査記録など |
| 失敗 | 重複、期限切れ、競合、部分失敗、権限変更など |
| 回復 | 再試行、取消、訂正、状態照会、補償処理など |

状態遷移はフロントエンドだけで制御しない。サーバー側でも現在状態と遷移条件を検証し、不正または順序外のリクエストを拒否する。OWASPもワークフローを明示的にモデル化し、フロントエンドの順序制御へ依存しないことを推奨している。[OWASP REST Security Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/REST_Security_Cheat_Sheet.html)

### 状態遷移ごとに定義する項目

```text
操作名:
開始状態:
終了状態:
実行可能な利用者・権限:
実行条件:
入力:
副作用:
監査記録:
失敗時の状態:
再試行・取消・訂正方法:
```

## 5. 認可とマルチテナント

認証済みであることと、特定の操作を許可されていることは別問題として扱う。

| 層 | 決めること | 例 |
| --- | --- | --- |
| 機能の発見可能性 | 機能の存在を表示するか | ナビゲーション、アップグレード・権限申請導線 |
| 現在の操作可否 | 現在の対象・状態で操作できるか | 無効化と理由表示 |
| サーバー側の強制 | リクエストを許可するか | API・データ層の認可 |

認可は原則として最小権限・デフォルト拒否とし、すべてのリクエストで検証する。対象IDを推測できても、別組織・別テナントのデータへアクセスできてはならない。[OWASP Authorization Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Authorization_Cheat_Sheet.html)

### 認可テスト

- URLを直接入力しても未許可画面・データへアクセスできない。
- APIへ直接リクエストしても拒否される。
- 画面表示後に権限が変更された場合、安全に回復できる。
- 一括操作に許可対象と不許可対象が混在する場合の挙動が定義されている。
- 別組織・別テナントの識別子を指定しても情報が漏れない。
- 管理者、一般利用者、閲覧専用など主要ロールを網羅している。

## 6. 一覧・検索・一括操作

一覧画面はデータの表示だけでなく、検索、比較、選択、操作を行う業務画面として設計する。IBM Carbonはデータテーブルを、並び替え、展開、単一・一括操作、ツールバー、検索、絞り込み、ページングを組み合わせるコンポーネントとして扱っている。[Carbon Data table](https://carbondesignsystem.com/components/data-table/usage/)

### 表示形式の選択

| 形式 | 適した用途 | 注意点 |
| --- | --- | --- |
| テーブル | 列を横断して値を比較する | HTMLの表構造を優先する |
| リスト | 行ごとに情報量・構造が異なる | 行操作と全体操作を分離する |
| データグリッド | セル移動・編集・選択が主要作業 | フォーカス移動とキーボード操作を実装する |

WAI-ARIAの`grid`は複合ウィジェットであり、実装者が内部のフォーカス移動を管理する。見た目が表であることだけを理由に使用しない。[WAI-ARIA Grid Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/grid/)、[WAI-ARIA Table Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/table/)

### 一覧画面の確認事項

- 検索条件、絞り込み、並び順、ページ、表示ビューを復元できる。
- 初期の並び順と同値時の順序を定義している。
- 件数が確定値か概算値かを区別している。
- 行単位、選択対象、一覧全体への操作を区別している。
- 欠損値、長い文字列、未知の値、削除済み参照を扱える。
- 詳細画面から戻った際の絞り込み、スクロール、選択状態を定義している。
- 読み込み中、データなし、検索結果なし、権限不足、取得失敗を区別している。

### 一括選択

- 「現在のページ」と「条件に一致する全件」を別の操作として表示する。
- 選択件数と操作対象を常に確認できるようにする。
- 条件変更時に選択を維持するか解除するかを定義する。
- 操作不能な対象を黙って除外しない。
- 部分失敗時は成功・失敗件数と失敗理由を表示する。
- 再試行では、すでに成功した対象を重複処理しない。

PatternFlyも、現在ページの選択と全ページの選択を区別し、選択件数を表示するパターンを示している。[PatternFly Bulk selection](https://www.patternfly.org/patterns/bulk-selection/)

## 7. フォームと編集

フォームは入力欄の集合ではなく、データを安全に変更する一連の処理として設計する。

| 段階 | 決めること |
| --- | --- |
| 読み込み | データの取得元、バージョン、権限、更新時刻 |
| 編集 | 入力検証、項目間の依存、未保存状態 |
| 送信 | 二重送信防止、処理中の操作、冪等性 |
| 競合 | 再読み込み、差分表示、統合、上書き可否 |
| 成功 | 保存対象、次の画面、フォーカス、通知 |
| 失敗 | 入力保持、項目別・全体エラー、再試行 |

### 保存方式の選択

| 条件 | 選択肢 |
| --- | --- |
| 軽微で容易に取り消せる変更 | 即時反映と取消操作を検討する |
| 複数項目の整合性が重要 | 明示的な保存操作を使用する |
| 影響が大きい変更 | 変更差分、対象、影響を確認する |
| 他の情報を参照しながら判断する | ダイアログではなくページやサイドパネルを検討する |
| 破壊的・不可逆な変更 | 操作名、対象、件数、結果、復旧可否を明記する |

入力エラーは項目の近くに表示し、ページ上部のエラー一覧から該当項目へ移動できるようにする。GOV.UK Design Systemもエラー一覧と各項目のエラーメッセージの併用を推奨している。[GOV.UK Error summary](https://design-system.service.gov.uk/components/error-summary/)、[Validation pattern](https://design-system.service.gov.uk/patterns/validation/)

不可逆な操作には明確な確認を設ける。ただし、ダイアログ外の情報を確認しなければ判断できない作業にはダイアログを使用しない。[Carbon Modal](https://carbondesignsystem.com/components/modal/usage/)、[Carbon Dialog pattern](https://v10.carbondesignsystem.com/patterns/dialog-pattern/)

## 8. 非同期処理と部分失敗

インポート、エクスポート、一括更新、レポート生成など、処理時間が長い操作は非同期ジョブとして扱うことを検討する。

### 非同期処理で表示する情報

- 受付済み、実行中、完了、部分完了、失敗を区別する。
- ジョブID、受付時刻、実行者、対象範囲、進捗を表示する。
- 画面を離れても進捗と結果を再確認できる。
- 通信タイムアウトと処理失敗を区別する。
- 同一リクエストの再送による重複処理を防止する。
- 失敗対象、理由、再試行可能性、結果の出力方法を提供する。
- 再試行を新しい処理として記録し、元の処理と関連付ける。

### 部分失敗

| 項目 | 表示・動作 |
| --- | --- |
| 集計 | 全件、成功件数、失敗件数 |
| 対象 | 失敗したレコードを特定できる |
| 理由 | 利用者が対応できる具体的な説明 |
| 再試行 | 失敗対象だけを再実行できる |
| 出力 | 必要に応じて失敗一覧をダウンロードできる |
| 監査 | 元処理と再試行の関係を追跡できる |

## 9. 画面状態とエラー

| 状態 | 表示する情報 | 主な操作 |
| --- | --- | --- |
| 初回読み込み | 読み込み対象、画面構造 | 必要に応じて取消 |
| バックグラウンド更新 | 表示値が古い可能性、操作可否 | 更新、状態確認 |
| データ表示 | データ範囲、取得元、更新時刻 | 主要業務操作 |
| データなし | 何が表示される場所か | 作成、設定、インポート |
| 検索結果なし | 結果を0件にした条件 | 条件の解除・修正 |
| 権限不足 | 不足権限、申請方法 | 権限申請、戻る |
| システムエラー | 失敗範囲、再試行の安全性 | 再試行、状態確認 |
| 部分成功 | 成功・失敗対象、理由 | 失敗分の再試行 |
| 競合・古いデータ | 更新者、更新時刻、差分 | 再取得、統合、再入力 |
| オフライン・タイムアウト | 入力保持、処理継続の有無 | 再接続、受付状態確認 |

空状態を一種類の共通表示へまとめない。Carbonも、データなし、検索結果なし、権限・システム・設定上の問題を区別し、それぞれ理由と次の操作を示している。[Carbon Empty states](https://carbondesignsystem.com/patterns/empty-states-pattern/)

### エラーメッセージ

- 何が起きたかを具体的に説明する。
- 利用者が修正できる場合は方法を示す。
- 再試行が安全かどうかを明示する。
- 入力値や選択状態を可能な限り保持する。
- 内部エラーコードやスタックトレースをそのまま表示しない。
- セキュリティ上秘匿すべき情報を含めない。

## 10. アクセシビリティ

WCAG 2.2 AAを基本的な基準とし、主要業務をキーボードと支援技術で完了できるようにする。[WCAG 2.2](https://www.w3.org/TR/WCAG22/)

### 実装・確認事項

- 適切なHTML要素を使用し、ARIAは必要な場合に限定する。
- すべての操作要素に認識可能な名前・役割・状態を持たせる。
- キーボードだけで主要フローを完了できる。
- フォーカス位置と移動順を予測できる。
- ダイアログ、エラー、動的更新後のフォーカスを管理する。
- 色だけで状態や重要度を伝えない。
- 200%拡大、リフロー、高コントラスト、モーション低減を確認する。
- エラーや処理結果をスクリーンリーダーへ通知する。
- 長いラベルや翻訳後の文言でも操作できる。

自動テストで検出できる問題は一部に限られるため、自動検査、手動確認、実際の支援技術による確認を組み合わせる。[Playwright Accessibility testing](https://playwright.dev/docs/accessibility-testing)

## 11. 国際化

- 日付のみの値と時刻を含む瞬間を別のデータ型として扱う。
- 保存時のタイムゾーンと表示時のタイムゾーンを区別する。
- 金額は数値と通貨コードを保持する。
- 数値、日付、通貨の入力・表示をロケールに合わせる。
- 並び替え、検索、文字比較のロケール・業務ルールを定義する。
- 人名を名・姓の固定構造として仮定しない。
- 翻訳による文言の伸長、右から左へ書く言語、複数形を確認する。

Unicode CLDRはロケール、数値・通貨、日付・時刻・タイムゾーン、照合順、人名に関する標準データを提供する。[Unicode LDML/CLDR](https://www.unicode.org/reports/tr35/)

## 12. 性能と信頼性

Core Web Vitalsを共通の基準として計測する。推奨値は、75パーセンタイルでLCP 2.5秒以内、INP 200ミリ秒以内、CLS 0.1以下である。[Web Vitals](https://web.dev/articles/vitals)

B2Bアプリケーションでは、業務固有の応答時間も計測する。

- 一覧の最初の有用な行が表示されるまでの時間。
- 検索・絞り込みから結果更新までの時間。
- キー入力からセル・フォームが応答するまでの時間。
- 一括処理の受付から追跡可能になるまでの時間。
- 大量データ表示時の描画量、メモリ、スクロール性能。
- タイムアウト後に処理状態を確認できるまでの時間。

### 信頼性の確認事項

- 書き込み操作の重複実行を防止する。
- 楽観的更新が失敗した場合に表示を戻せる。
- 再試行可能な操作と不可能な操作を区別する。
- 古いデータによる上書きを検知する。
- オフライン・低速回線でも入力を失わない。
- 実験環境だけでなく実利用環境の性能を計測する。

## 13. デザインシステムと実装規約

共通コンポーネントには、見た目だけでなく使用条件、非推奨条件、状態、アクセシビリティ、テストを含める。

| 項目 | 定義内容 |
| --- | --- |
| 用途 | どの問題を解決するか |
| 使用条件 | いつ使用するか、いつ使用しないか |
| API | プロパティ、イベント、型、初期値 |
| 状態 | 通常、無効、読み込み、エラー、空など |
| 組み合わせ | 他コンポーネントとの配置・優先順位 |
| アクセシビリティ | キーボード、読み上げ、フォーカス |
| 文言 | ラベル、エラー、状態メッセージ |
| テスト | 描画、操作、視覚、アクセシビリティ |

デザイン値は色や余白の生値を画面ごとに追加せず、目的を表すデザイントークンで管理する。Design Tokens Community Groupの仕様は、デザイン上の決定をツール非依存で交換する形式を定義している。[Design Tokens Format Module 2025.10](https://www.designtokens.org/TR/2025.10/format/)

Storybookなどを使用する場合は、通常状態だけでなく、読み込み、空、エラー、権限不足、長文、キーボード操作などを独立した使用例として管理する。[Storybook UI testing](https://storybook.js.org/docs/writing-tests)

## 14. テスト戦略

| テスト層 | 主な対象 |
| --- | --- |
| 単体テスト | 状態遷移、入力検証、選択、権限結果の表示 |
| コンポーネントテスト | 各状態の描画、操作、アクセシビリティ |
| API契約テスト | 型、エラー、認可、冪等性、競合 |
| 結合テスト | フォーム、一覧、非同期処理の連携 |
| E2Eテスト | 主要業務フロー、権限、復旧、監査 |
| 視覚回帰 | レイアウト、状態表示、主要コンポーネント |
| 性能テスト | 大量データ、応答時間、長時間操作 |
| セキュリティテスト | 認可、テナント境界、直接APIアクセス |

### 代表的なテストケース

- 大量データで検索、絞り込み、並び替え、ページングを行う。
- 現在ページの選択と条件一致全件の選択を区別する。
- 一括操作に操作可能・不可能な対象が混在する。
- データなしと検索結果なしを区別する。
- 編集中または送信時に権限が変更される。
- 他の利用者が同じデータを先に更新する。
- 一括処理の一部だけが成功する。
- 通信タイムアウト後に処理の受付状態が不明になる。
- インポートデータの一部の行だけにエラーがある。
- 破壊的操作の対象・影響・復旧可否を確認する。
- キーボードだけで主要業務を完了する。
- 長い翻訳文、複数通貨、異なるタイムゾーンを表示する。

視覚回帰は見た目の変化を検出できるが、業務上の意味や認可は検証できない。DOM、API、監査記録の検証と組み合わせる。[Playwright Visual comparisons](https://playwright.dev/docs/test-snapshots)

## 15. 監査とオブザーバビリティ

### 監査記録

- 実行者、所属組織、操作、対象、実行時刻を記録する。
- 変更前後の値または追跡可能な差分を保持する。
- 操作理由、承認者、関連ジョブを必要に応じて記録する。
- 再試行、取消、訂正を元の操作と関連付ける。
- 監査記録自体へのアクセスも認可する。
- 個人情報、秘密情報、不要な入力内容をログへ残さない。

### フロントエンド計測

- 主要業務の完了率と所要時間。
- 入力・検証・認可・通信・競合エラーの分類。
- 再試行と回復の成功率。
- 検索・絞り込み・一括操作の利用状況。
- 実利用環境の応答時間とWeb Vitals。
- クライアントイベント、APIリクエスト、非同期ジョブを相関できる識別子。

計測値だけで操作内容や個人情報が推測されないよう、収集目的、項目、保存期間、閲覧権限を定義する。

## 16. 設計・実装レビュー

### 業務

- 用語、状態、操作可能条件が業務ルールと一致している。
- 正常系だけでなく、例外・部分失敗・競合を扱っている。
- 対象範囲、件数、変更結果を利用者が確認できる。
- 取消、訂正、再試行などの回復方法がある。

### 権限・セキュリティ

- サーバー側で全リクエストを認可している。
- 組織・テナント境界を越えてデータへアクセスできない。
- 画面表示後の権限変更を扱える。
- エラーメッセージやログから機密情報が漏れない。

### 操作・情報設計

- 頻繁な業務を少ない操作で完了できる。
- 一覧の検索条件・並び順・ページを復元できる。
- 行、選択対象、一覧全体への操作を区別している。
- 読み込み、空、検索結果なし、権限不足、障害を区別している。

### 品質

- 主要フローをキーボードで完了できる。
- 自動検査と手動アクセシビリティ確認を実施している。
- 大量データ・低速回線・長文・異なるロケールを確認している。
- 単体、コンポーネント、API、E2E、視覚回帰の責務が分かれている。
- 本番環境で性能、失敗、回復を観測できる。
